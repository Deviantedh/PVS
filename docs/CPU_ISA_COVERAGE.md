# CPU ISA Coverage
## ARM Cortex‑M3 (ARMv7‑M, Thumb/Thumb‑2) — план покрытия инструкций

Версия: 1.0  
Статус: Draft (MVP + Roadmap)

> Цель документа — зафиксировать **контракт по ISA**: что поддерживаем в MVP, что планируем, какие упрощения допустимы, и как это тестируем.  
> Документ нужен для параллельной работы ИИ‑агентов: каждый блок можно реализовывать независимо, имея чёткие критерии готовности.

---

## 0. Базовые принципы исполнения

### 0.1 Endianness
- **Little Endian** (фиксировано).

### 0.2 Размер инструкций
- Поддерживаем **Thumb‑2**: инструкции **16‑бит** и **32‑бит**.
- Fetch должен уметь корректно читать halfword’ы из Flash/RAM и формировать 32‑бит слово инструкции.

### 0.3 Модель времени (MVP)
- 1 инструкция = 1 tick (см. `ARCHITECTURE.md`).

### 0.4 Политика выравнивания (фиксируем в MVP)
Выбираем режим **STRICT** для MVP:
- Любой unaligned доступ для LDR/STR/LDM/STM/PUSH/POP → **BusFault → HardFault** (агрегируем).
- Причина: упрощение, детерминизм и тестируемость.

> После MVP можно перейти в режим “Cortex‑M3‑like” и поддержать часть unaligned, но это отдельная задача.

### 0.5 Ошибки исполнения
- Неподдержанная инструкция → `STOP_UNSUPPORTED_INSTR` + диагностика (PC, word, mnemonic если возможно).
- Ошибка декодирования → `STOP_UNSUPPORTED_INSTR`.
- Доступ к unmapped MMIO → `STOP_FAULT`.

## 1. Уровни соответствия (phased plan)

### L1 — MVP (обязательно)
Достаточно, чтобы:
- стартовать с reset vector
- дойти до `main()`
- работать с простыми обработчиками IRQ (TIM2/USART1)
- печатать в UART

### L2 — Расширение (желательно в семестре)
Достаточно, чтобы:
- запускать “реалистичные” прошивки (больше оптимизаций компилятора)
- обслуживать IT blocks / CBZ / CBNZ
- покрыть большую часть Thumb‑2 data processing

### L3 — “Почти полный ARMv7‑M” (после семестра)
- остальное из ARMv7‑M (включая редкие инструкции/особые кейсы), либо документированное ограничение.

## 1.1 Минимальный набор инструкций для запуска C‑прошивки

Ниже — практический “минимум”, без которого типичная bare‑metal C‑прошивка для Cortex‑M3 не стартует (reset handler → `main()`), либо не сможет вызвать/вернуть функции и работать со стеком.

### 1.1.1 Минимум для `Reset_Handler` и входа в `main()`
Обычно требуется:

- **Переходы и вызовы**: `B`, `BL`, `BX`
- **Работа со стеком**: `PUSH`, `POP` (или эквивалент через `STM/LDM`)
- **Загрузка констант/адресов**: `LDR (PC‑relative literal)`
- **Базовые арифметика/перемещения**: `MOV/MOVS`, `ADD/ADDS`, `SUB/SUBS`
- **Сравнения/ветвления**: `CMP`, `B<cond>`

### 1.1.2 Минимум для простого кода без оптимизаций (`-O0`)
Для `-O0` компилятор чаще генерирует “прямолинейные” последовательности, но почти всегда встречаются:

- `LDR/STR` (word) с immediate offset
- `LDRB/STRB` для работы с `char*` и UART‑буферов
- `LSL/LSR/ASR` (особенно для адресной арифметики и масок)
- `AND/ORR/EOR` (маски регистров периферии)
- `NOP` (иногда для выравнивания/задержек)

### 1.1.3 Что обычно появляется при оптимизациях (`-O1/-O2`)
При оптимизациях повышается шанс появления:

- `CBZ/CBNZ`
- `IT` + условные инструкции внутри блока
- `ADC/SBC` (например, в многоразрядной арифметике/копировании)
- `LDM/STM` (оптимизация прологов/эпилогов)
- `MUL` (в вычислениях даже без явного умножения в исходнике)

**Рекомендация для MVP:** первые тест‑прошивки собирать с `-O0`, чтобы отложить IT/CBZ/LDM/STM.

### 1.1.4 Минимум для IRQ‑обработчиков
Для работы простых ISR (TIM2/USART1) требуется всё из 1.1.1 +:

- корректный **exception entry/return** (stack frame)
- `CPSID/CPSIE` (опционально, если ISR/код использует маскирование)

## 10. Что точно вне scope (для MVP/семестра)

- Полная точность таймингов (cycle‑accurate).
- FPU инструкции (в Cortex‑M3 их нет).
- JTAG/DWT/ITM полная модель (можно заглушки).
- Полная модель памяти STM32 (FLASH programming, option bytes).

## 11. Критерии готовности ISA‑слоя

### 11.1 MVP (L1) считается выполненным, если:
1) Прошивка “hello_uart” запускается от reset vector и пишет строку в USART1.
2) Прошивка “timer_irq” принимает IRQ от TIM2 и выполняет ISR (увеличение счётчика/печать).
3) Симулятор умеет корректно завершаться по лимиту инструкций и отдаёт диагностику.
4) Неподдержанная инструкция приводит к контролируемой остановке (не segfault).

### 11.2 Требования к UT (unit tests)
- Для каждой реализованной инструкции: минимум 3 теста:
  - базовый кейс
  - кейс на флаги (если применимо)
  - corner case (переполнение/перенос/нулевое значение)

## 12. План раздачи задач ИИ‑агентам (рекомендуемый)

Разбивать задачи по “вертикальным слайсам”:

1) Fetch/decode 16‑бит + B/BL/BX + MOV/ADD/SUB + LDR literal  
2) PUSH/POP + стек + вызов/возврат  
3) LDR/STR word + memory/bus faults  
4) Флаги APSR: N/Z/C/V для ADD/SUB/CMP  
5) Conditional branches + condition evaluation  
6) (L2) Byte/Halfword + CBZ/CBNZ + IT blocks

Каждый слайс должен поставляться с UT и минимум одной FW‑прошивкой.

## 13. Дополнительные рекомендации для сборки тестовых прошивок

Для упрощения реализации и тестирования рекомендуются следующие настройки компилятора и линковщика:

- Использовать флаги: `-mcpu=cortex-m3 -mthumb -O0` для минимизации сложных инструкций и упрощения кода.
- Включать `-ffreestanding` и `-fno-builtin` для отключения использования стандартных библиотек и встроенных функций, чтобы избежать неожиданных инструкций.
- Избегать использования IT-блоков и сложных оптимизаций на ранних этапах.
- Использовать минимальный runtime без newlib или других libc, чтобы максимально контролировать генерируемый код.
- Компилировать и линковать с явным указанием стартового файла и минимального crt0, чтобы обеспечить корректный старт и минимальный набор инструкций.

Такой подход позволит постепенно расширять поддержку ISA, минимизируя риски и облегчая отладку.

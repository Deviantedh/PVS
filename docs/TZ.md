Техническое задание (ТЗ)

Проект: Симулятор STM32F103C8T6 (Blue Pill) + микросервис управления

0. Термины и определения
	•	SoC/MCU — микроконтроллер STM32F103C8T6.
	•	Симулятор — программа, исполняющая бинарный код STM32 в виртуальной модели памяти и периферии.
	•	Ядро (core) — модель CPU ARM Cortex-M3 (ARMv7-M, Thumb/Thumb-2).
	•	MMIO — memory-mapped I/O, периферия как регистры в адресном пространстве.
	•	NVIC — контроллер прерываний ARM.
	•	KeyDB — брокер/хранилище для обмена задачами/результатами (совместим с Redis-протоколом).
	•	OTel — OpenTelemetry, система трассировки/логирования.
	•	MVP — минимально жизнеспособная версия, допускающая неполноту, но демонстрирующая архитектуру и работоспособность  ￼.

⸻

1. Введение

1.1 Актуальность

Проект позволяет изучить:
	•	устройство STM32F103,
	•	систему команд ARMv7-M/Thumb-2,
	•	работу периферии (таймер, UART, прерывания),
	•	подходы к проектированию, тестированию и документированию в командной разработке  ￼.

1.2 Цель

Создать симулятор микроконтроллера STM32F103C8T6 “как на плате Blue Pill”, способный исполнять код и моделировать выбранные периферийные блоки, а также микросервис (Go) для приема задач и выдачи результатов выполнения  ￼.

1.3 Задачи
	1.	Реализовать ядро симулятора на C (CPU + память + MMIO).  ￼
	2.	Реализовать периферию: таймер, UART, контроллер прерываний (и базовые порты ввода-вывода как минимум на уровне MMIO-заглушек).  ￼
	3.	Реализовать микросервис на Go: прием кода через KeyDB, выдача результата через KeyDB, логирование/трейсинг через OpenTelemetry.  ￼
	4.	Организовать тестирование (unit + интеграционные тесты на прошивках).  ￼
	5.	Подготовить пояснительную записку/отчет с разделами: ТЗ, архитектура, реализация, тестирование и т.д.  ￼

⸻

2. Назначение разработки

Система предназначена для:
	•	выполнения (симуляции) программ, собранных под STM32F103C8T6, без реального железа;
	•	автоматизированного прогона тестовых прошивок в “облачном/серверном” окружении;
	•	сбора артефактов выполнения (логи UART, статус завершения, метрики/трейсы) для анализа.

⸻

3. Требования к системе (функции, состав, ограничения)

3.1 Состав системы

Система состоит из двух основных компонентов  ￼:
	1.	Ядро симулятора (C)
	•	исполняет код STM32
	•	моделирует периферийные блоки (таймер, UART, NVIC/прерывания, порты I/O)
	2.	Микросервис (Go)
	•	принимает исполняемый код через KeyDB (формат уточняется/фиксируется в рамках проекта)
	•	запускает симулятор
	•	публикует результаты выполнения через KeyDB
	•	логирование на базе OpenTelemetry

3.2 Роли и “заинтересованные лица”
	•	Преподаватель/заказчик: ожидает работающий MVP, архитектуру, тесты, документацию.
	•	Команда разработки: удобная сборка, отладка, понятные интерфейсы.
	•	ИИ-агенты: должны получать задачи “по модулю” с четкими входами/выходами и тест-критериями (см. раздел 8).

⸻

4. Функциональные требования (ядро симулятора на C)

4.1 Модель памяти и адресное пространство

Симулятор должен реализовать:
	•	Flash region (начальная загрузка прошивки, векторная таблица)
	•	SRAM region
	•	MMIO region периферии
	•	System Control Space (0xE000E000…) для NVIC/SCB/SysTick (минимально для прерываний)

Минимальная загрузка (Reset):
	•	MSP берется из *(uint32_t*)FLASH_BASE
	•	PC берется из *(uint32_t*)FLASH_BASE + 4
	•	поддержка VTOR (SCB) — желательно; допускается ограниченный режим, но это должно быть явно описано.

4.2 CPU core (ARM Cortex-M3, Thumb/Thumb-2)

Требование из лекции: “ядро процессора с возможностью исполнения всех ассемблерных команд”  ￼.
Чтобы сделать это реализуемым в семестр, вводим уровни соответствия:

Уровень L1 (MVP, обязательно):
	•	корректный fetch/decode/execute для 16/32-бит Thumb-2
	•	базовые инструкции, достаточные для выполнения C-прошивки:
	•	data processing (ADD/SUB/MOV/CMP/AND/ORR/EOR, shifts)
	•	load/store (LDR/STR, byte/halfword, PUSH/POP, LDM/STM минимум)
	•	ветвления (B/BL/BX, условные B)
	•	инструкции для взаимодействия с исключениями: SVC, BKPT (как ловушки), CPS (минимально), NOP/WFI (можно упростить)
	•	флаги APSR (N,Z,C,V) обновляются корректно для реализованных инструкций.

Уровень L2 (расширение, желательно):
	•	IT-блоки, CBZ/CBNZ
	•	UMULL/SMULL и прочие редко используемые (по приоритету)
	•	исключительные/непредсказуемые случаи (UNPREDICTABLE) фиксируются как “trap + лог”, допускается упрощение.

Явная политика: если “полное покрытие ISA” не успеваем, проект обязан:
	•	иметь таблицу покрытия инструкций (implemented / planned / not supported),
	•	уметь детектировать неподдержанную инструкцию и завершать выполнение контролируемо (с кодом ошибки и диагностикой).

4.3 Исключения и прерывания

Обязательно:
	•	реализация exception entry/return (stacking/unstacking базового фрейма Cortex-M)
	•	поддержка NVIC для внешних IRQ (см. 5.2)
	•	поддержка хотя бы системных исключений: Reset, HardFault (как fallback), SysTick (опционально), SVC (для тестов)

⸻

5. Функциональные требования (периферия STM32)

5.1 Таймер (TIMx)

Минимальный блок: TIM2 (или другой выбранный в ТЗ команды — фиксируем один).
Поддержать регистры (MVP):
	•	CR1 (CEN)
	•	PSC
	•	ARR
	•	CNT
	•	DIER (UIE)
	•	SR (UIF)

Поведение:
	•	таймер увеличивает CNT по внутренним “тикам симуляции”
	•	при переполнении устанавливает UIF; при UIE=1 поднимает IRQ таймера через NVIC.

5.2 Контроллер прерываний (NVIC)

MVP:
	•	enable/disable (ISER/ICER)
	•	pending set/clear (ISPR/ICPR)
	•	приоритеты (IPR, хотя бы старшие 4 бита)
	•	выбор “следующего” IRQ по приоритету и pending/enabled
	•	корректная интеграция с exception entry/return.

5.3 UART/USART

Минимальный блок: USART1 (часто используется на Blue Pill).
MVP-регистры:
	•	SR (TXE, RXNE)
	•	DR
	•	BRR (можно хранить без точной симуляции таймингов)
	•	CR1 (UE, TE, RE, RXNEIE, TXEIE)

Ввод/вывод:
	•	TX: запись в DR формирует байт/строку в выходной поток (лог/буфер результата)
	•	RX: симулятор может получать входные байты из “input payload” задачи; установка RXNE и IRQ при RXNEIE.

5.4 Порты ввода-вывода (GPIO)

Лекция упоминает порты ввода-вывода как часть блоков  ￼. Для семестрового MVP:
	•	допускается минимальная заглушка MMIO, чтобы прошивки не падали при чтении/записи регистров GPIO/RCC/AFIO,
	•	плюс один-два реально работающих сценария (например, чтение входного пина из заданного входа).

⸻

6. Функциональные требования (микросервис на Go)

6.1 Прием задач через KeyDB

Требование: прием исполняемого кода через KeyDB, формат уточняется  ￼.
В рамках проекта формат фиксируем так:

Объект задачи (логический контракт):
	•	job_id (строка/UUID)
	•	firmware (байтовый массив: raw .bin или .elf — выбирается проектом; MVP проще .bin)
	•	config:
	•	лимит инструкций/времени симуляции
	•	начальный ввод UART (опционально)
	•	режим логирования (уровень)
	•	опционально: “карта” входов GPIO
	•	expected (опционально для автотестов): ожидаемый вывод UART/код завершения

Механизм очереди (MVP):
	•	BLPOP/BRPOP по ключу очереди, либо streams — выбирается командой.
	•	Повторная доставка/ack (если streams) — желательно, но можно отложить.

6.2 Запуск симулятора

Микросервис обязан:
	•	сохранить прошивку во временный файл/передать через stdin (фиксировать один способ)
	•	запустить C-симулятор как отдельный процесс (preferred) либо как библиотеку через cgo (необязательно)
	•	контролировать таймаут/лимит шагов
	•	собрать артефакты выполнения.

6.3 Выдача результата через KeyDB

Результат включает:
	•	job_id
	•	status (OK / TIMEOUT / CRASH / UNSUPPORTED_INSTR / FAULT)
	•	exit_code (число)
	•	uart_stdout (строка/байты)
	•	trace_summary (опционально: число инструкций, последнее PC, причина остановки)
	•	ссылки/ключи на большие артефакты (если нужно)

6.4 Логирование и трассировка (OpenTelemetry)

Требование: система логирования на базе OpenTelemetry  ￼.
MVP:
	•	трасса на каждый job_id
	•	спаны: dequeue_job, prepare_payload, run_simulator, publish_result
	•	атрибуты: job_id, status, instr_count, timeout_ms
	•	экспорт в stdout (OTLP/console) или в локальный collector (как договоритесь в команде)

7. Нефункциональные требования

7.1 Платформы и сборка
	•	Код редактируется в VS Code (как в лекции)  ￼
	•	Поддержка Linux/macOS через CMake (или Meson) + Go modules
	•	Желательно Docker-окружение для воспроизводимой сборки (опционально)

7.2 Надежность (в рамках MVP)
	•	Симулятор может “падать” в редких случаях, но должен уметь:
	•	возвращать диагностируемые коды ошибок,
	•	писать понятный лог (последние N инструкций/PC/регистры),
	•	не зависать бесконечно (лимит шагов/таймаут обязателен).

7.3 Производительность

MVP ориентирован на корректность и архитектуру. Тем не менее:
	•	симулятор должен выполнять простые тест-прошивки за секунды, а не минуты (на обычном ноутбуке).

7.4 Документируемость
	•	Документация в исходниках + Markdown (лекция упоминает Markdown/Obsidian)  ￼
	•	В репозитории должны быть: README, SPEC/TZ, ARCHITECTURE, TESTING, CONTRIBUTING.

⸻

8. Требования к разработке с ИИ-агентами (процесс)

Лекция прямо фиксирует, что основная часть кода пишется ИИ-агентами, но вы должны понимать и уметь ловить ошибки ИИ  ￼  ￼.

8.1 Правила постановки задач агентам

Каждая задача агенту оформляется как “мини-спецификация”:
	•	цель (что сделать)
	•	входы/выходы (API/функции/формат данных)
	•	критерии готовности (тесты/сценарии)
	•	ограничения (что не трогать, какие файлы модифицировать)

8.2 Политика ревью
	•	любой PR от агента проходит человеческую проверку: корректность, стиль, тесты
	•	обязательные sanity-checks: компиляция + unit tests

10. Тестирование и критерии приемки

10.1 Тесты симулятора (C)

Unit tests:
	•	декод/исполнение отдельных инструкций
	•	NVIC: pending/enable/priority selection
	•	TIM2: overflow + IRQ
	•	USART1: TX/RX flags + IRQ

Integration tests (прошивки):
	•	прошивка “hello uart”
	•	прошивка “timer irq increments counter + prints”
	•	прошивка “uart echo”

10.2 Тесты микросервиса (Go)
	•	тест парсинга/валидации job payload
	•	тест таймаута/лимита шагов
	•	тест публикации результата
	•	тест OTel: создание trace/span + атрибуты (минимально smoke)

10.3 Критерии приемки MVP

MVP считается готовым, если:
	1.	Микросервис принимает задачу из KeyDB и возвращает результат в KeyDB  ￼
	2.	Симулятор выполняет хотя бы 3 эталонные тест-прошивки (UART, таймер+IRQ, базовый “main”)
	3.	NVIC+таймер+UART работают в сценариях прерываний
	4.	Есть документация: ТЗ + архитектура + описание тестов/запуска  ￼

12. Открытые решения (нужно зафиксировать командой в начале)

Чтобы ТЗ стало “замкнутым”, команде нужно выбрать и записать в TZ.md:
	1.	Формат прошивки: .bin или .elf (MVP проще .bin)
